# pWPS Agent 产品需求文档 requirements.md

## 1. 文档定位

本文档定义 pWPS Agent 的产品目标、用户场景、MVP 范围、核心能力、非目标与成功指标。它回答“要做什么、为什么做、第一阶段做到什么程度”，不展开具体代码结构和详细伪代码。

---

## 2. 产品定位

pWPS Agent 是一个面向焊接工艺设计场景的智能草案生成系统。

它通过大模型、知识检索、字段状态机、证据链和人机协同，帮助用户从不完整需求出发，逐步形成一份可追踪、可解释、可审查的 pWPS 草案。

一句话定位：

> pWPS Agent 是一个字段驱动、证据支撑、人机协同、可审计的 pWPS 草案生成系统。

---

## 3. 背景问题

直接让大模型根据一句话生成完整 pWPS，容易出现：

1. 缺失关键信息时强行生成。
2. 编造项目编号、客户、公司、标准依据等字段。
3. 焊法、焊材、参数、热处理之间不一致。
4. 检索结果没有明确服务于具体字段。
5. 生成内容完整但缺少证据链和风险说明。
6. Auto 和 Guided 被实现成两套割裂流程。
7. Agent 自由规划过多，流程不稳定、难复现。

因此，本系统不以“自由 Agent 聊天”为核心，而以“字段组确认工作流”为核心。

---

## 4. 核心设计判断

Auto Draft 与 Guided Draft 不是两套工作流，而是一套字段确认工作流的两种决策对象。

```text
Auto Draft   -> VirtualDecisionActor
Guided Draft -> HumanDecisionActor
```

底层流程统一：

```text
输入理解
  -> 字段映射
  -> 字段组队列
  -> 规划
  -> 检索
  -> 证据归一化
  -> 候选生成
  -> 决策确认
  -> 反向评估
  -> 全局审计
  -> 结构化输出
```

---

## 5. 目标用户

### 5.1 焊接工程师

希望系统辅助生成 pWPS 草案、推荐字段、查找依据、指出风险。

### 5.2 工艺技术人员

用于初步工艺方案设计、方案比选、资料准备。

### 5.3 项目技术负责人

关注草案是否具备可审查性、是否有风险说明、是否可以作为技术沟通材料。

### 5.4 研发与知识工程人员

维护字段体系、标准文档、知识库、Prompt、Skill、检索策略和评估集。

## 6. 核心场景

### 场景一：快速生成草案

用户输入：

```text
Q345R，12mm，对接焊，平焊，GMAW，帮我生成一份 pWPS 草案。
```

系统进入 Auto Draft 模式，VirtualDecisionActor 基于候选和证据做谨慎选择，输出一份标记风险和待确认字段的草案。

### 场景二：逐步引导生成

用户选择 Guided Draft。系统按照字段组展示推荐值、候选值、证据、风险，用户逐步确认。

### 场景三：用户覆盖系统建议

系统推荐 ER50-6，用户提出使用 J422。系统发现其与当前 GMAW 焊接方法不匹配，提示如坚持 J422 应同步调整焊接方法为 SMAW。

### 场景四：全局审计发现冲突

字段确认完成后，审计发现焊丝直径、电流范围、热输入不一致，生成 repair target，重新进入参数字段组确认。

## 7. 功能范围

### 7.1 必须支持

1. 自然语言输入。
2. 字段抽取与标准化映射。
3. 字段组确认队列。
4. Auto Draft 与 Guided Draft 的统一抽象。
5. 字段候选生成。
6. 证据归一化与字段挂载。
7. 用户覆盖反向评估。
8. 全局审计。
9. 结构化输出。
10. 前端字段确认工作台。
11. 多端适配。
12. 运行状态、讨论记录、证据链可追踪。

### 7.2 非目标

第一阶段不做：

1. 正式可签发 WPS。
2. 替代 PQR/WPQR 工艺评定。
3. 企业审批流、签章流、正式文件归档。
4. 完整企业知识库治理平台。
5. 一次性覆盖全部焊接方法和全部材料体系。
6. 复杂权限系统。
7. 微服务拆分。

---

## 8. MVP-0 范围

MVP-0 目标是跑通 Auto CLI 最小闭环。

必须包含：

```text
自然语言输入
  -> 字段抽取
  -> 固定字段队列
  -> 简单知识检索 / model prior
  -> 候选生成
  -> VirtualDecisionActor
  -> 简单审计
  -> pwps.json / field_report.json / risk_report.json
```

MVP-0 不包含：

```text
完整 Web UI
完整 Guided interrupt/resume
Qdrant
MinIO
复杂文档解析
复杂 repair loop
完整企业知识库
```

---

## 9. 关键字段组

MVP 至少覆盖：

1. `basic_condition_group`：母材、厚度、焊法、接头形式、焊接位置。
2. `consumable_group`：焊材、焊材规格、保护气体。
3. `parameter_group`：电流、电压、速度、热输入、极性。
4. `thermal_group`：预热、层间温度、PWHT。
5. `meta_group`：文档元信息，只能用户提供或留空。

---

## 10. 输出要求

系统最终优先输出结构化数据，而不是直接生成自然语言文档。

推荐输出：

```text
pwps.json
field_report.json
evidence_report.json
risk_report.json
discussion_trace.json
render_payload.json
```

文档渲染由独立 Render 模块完成。

---

## 11. 成功指标

### 功能指标

1. 用户提供最小启动字段后，系统可以生成 pWPS 草案。
2. Auto 和 Guided 共用同一字段确认流程。
3. 每个关键字段都有来源、候选、证据和风险说明。
4. 全局审计能发现明显冲突。
5. 输出 JSON 可被渲染模块直接使用。

### 质量指标

1. 禁止推断字段不得被模型伪造。
2. 焊法与焊材明显不匹配应被检测。
3. 高风险字段必须有风险提示。
4. 用户覆盖系统建议必须记录。
5. Auto 输出必须标记待人工确认字段。

### 体验指标

1. Guided 模式中用户不需要理解底层检索细节。
2. 字段确认卡片应清晰展示推荐理由。
3. 移动端、平板端、桌面端都可完成核心流程。
4. 用户可以快速接受推荐，也可以展开证据深入查看。

---

## 12. 工程约束

1. 前端使用 pnpm。
2. 后端使用 uv。
3. 字段、状态、API、Skill schema 使用 Pydantic v2。
4. 状态使用枚举，禁止裸字符串状态。
5. 新技术引入前必须调查最佳实践。
6. 修改代码前必须分析影响面。
7. 测试必须保护核心路径，但避免过度测试。
