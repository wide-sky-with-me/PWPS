# pWPS Agent 工程设计与开发治理准则 engineering_principles.md

## 1. 文档定位

本文档约束工程设计、技术引入、代码修改、测试策略和长期可维护性。

目标是避免碎片化开发，避免用临时 patch 解决局部问题却引入长期包袱。

---

## 2. 核心原则

1. 先理解问题边界，再写代码。
2. 先确认设计影响，再修改实现。
3. 先复用现有抽象，再新增抽象。
4. 先扩大已有模块能力，再引入新模块。
5. 先调查最佳实践，再引入新包或新技术。
6. 解决问题不能只看当前报错，还要看对架构、状态、测试和用户体验的影响。
7. 不为一次性需求制造长期复杂度。
8. 不用临时 patch 绕过领域规则、状态机或审计规则。

---

## 3. 新包 / 新技术引入规则

引入前必须回答：

```text
1. 这个包解决什么具体问题？
2. 现有技术栈是否已经能解决？
3. 替代方案有哪些？
4. 是否影响前端包体、后端依赖复杂度或部署复杂度？
5. 是否有活跃维护和清晰文档？
6. 是否与当前技术栈兼容？
7. 是否需要额外测试、迁移或运维服务？
8. 如果未来移除，成本有多高？
```

---

## 4. 修改影响分析

修改前必须检查是否影响：

```text
FieldSpec / FieldState
WorkflowState
DecisionActor
Skill schema
AuditRule
KnowledgeService
API contract
Database migration
Frontend state
Tests
Docs
```

---

## 5. 避免碎片化开发

禁止：

```text
到处加 if patch
复制一份相似逻辑
为一个页面临时创建一套独立状态结构
绕过已有 schema
绕过 Field Registry
绕过 Audit Engine
```

优先：

```text
补全原有抽象
复用已有服务
扩展已有 schema
补充规则而不是绕过规则
```

---

## 6. 测试治理

原则：

1. 测核心，不测噪声。
2. 能合并则合并。
3. 过时测试及时删除。
4. 不为覆盖率写脆弱测试。
5. 不测试第三方库行为。
6. LLM 测试优先使用 mock / fixture。

---

## 7. 文档同步

以下变更必须更新文档：

```text
API 变更
Schema 变更
Workflow 变更
Skill 输入输出变更
新增依赖
新增服务
开发阶段边界变化
安全策略变化
```

## Agent 指令治理原则

开发规则同样需要遵守软件设计原则。`AGENTS.md` 不应不断累积细节规则，而应作为精简路由入口。

规则新增时先判断：

```text
是否每次开发都必须遵守？
  是 -> 放入 AGENTS.md 常驻规则
  否 -> 放入对应专项文档，并更新 AGENTS.md 路由表
```

禁止用复制粘贴方式在多个文档散布同一规则。需要共享的规则应放在最权威的文档中，其他文档只引用。
